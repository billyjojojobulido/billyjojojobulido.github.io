算法与数据结构 - 数组篇(其四): 分治法
August 08, 2024
1.5k words 15 mins

简述

分治，字面上的解释是「分而治之」，就是把一个复杂的问题分成两个或更多的相同或相似的子问题， 直到最后子问题可以简单的直接求解，原问题的解即子问题的解的合并。

分治的过程

分治算法的核心思想就是「分而治之」。

因此分治的大致流程也可以分为三步: 分解 -> 解决 -> 合并

  1. 分解原问题为结构相同的子问题。
  2. 分解到某个容易求解的边界之后，进行递归求解。
  3. 将子问题的解合并成原问题的解。

适合用"分治"的特征

从分治法的流程反推，我们可以发现适合用分治法解决的问题应该具备以下特征:

  1. 问题的规模缩小到一定的程度就可以容易地解决。
  2. 问题可以分解为若干个规模较小的相同问题(最优子结构性质)。利用子问题的解可以合并为整体问题的解。
  3. 题所分解出的各个子问题是相互独立的。

都想到这里了，索性我们再多想一层，当问题具备以下特征时，应该尽量避免用分治法

  • 当问题存在较高的维度时

    • 这一特征同时违背了原则1和原则3。譬如图的连通性问题，当图论问题具有较高维度或者多层次的关系时，因为图中复杂的点、边关系，很难拆解成独立的子问题。

  • 当问题涉及全局优化时

    • 这一特征违背了原则2。以旅行商问题(TSP)问题为例，目标是找到经过所有城市的最短路径，那这种问题要考虑所有城市的路径组合。 这种情况下，无论怎么拆分子问题，最细粒度的子问题也无非是考虑要不要从A城市出发去B城市，那么很显然每个子问题的解可能会影响全局解，因此子问题的解好求， 但会陷进局部的优化问题，从而破坏了全局一致性，导致无法合并出全局最优解。

  • 当子问题具有高度依赖时

    • 这一特征违背了原则3。譬如,以0/1背包问题为例，虽然确实可以用分治法分解成子任务(一次只考虑一个物品是否放入背包)。 但是子问题会依赖其他子问题的解(其他物品的放入情况)，那么这种问题显然更适合用动态规划求解。

这些问题通常需要使用其他算法和策略，如回溯、贪心算法、动态规划、启发式算法等。解决这些问题时，通常需要综合考虑问题的整体结构和特性，而不仅仅是分解问题。

分治法解归并

归并排序可以说是最典型的分治题型。大体的分治思路如代码所示:

def merge_sort(一个数组) {
    if (可以很容易处理) return;
    merge_sort(左半个数组);
    merge_sort(右半个数组);
    merge(左半个数组, 右半个数组);
}

因为分治算法的套路是 分解 -> 解决（触底）-> 合并（回溯），先左右分解，再处理合并，回溯就是在退栈，即相当于后序遍历。

减治思想

「分治思想」的特例是「减治思想(Decrease-and-Conquer)」: 每一步将问题转换成为规模更小的子问题。「减治思想」思想的典型应用是「二分查找」「选择排序」「插入排序」「快速排序」算法。 「分治」与「减治思想」的区别如下:

  • 分治思想: 将一个问题拆分成若干个子问题，然后再逐个求解，根据各个子问题得到的结果得到原问题的结果
  • 减治思想: 在拆分子问题的时候，只将原问题转化成 一个 规模更小的子问题，因此子问题的结果就是上一层原问题的结果， 每一步只需要解决一个规模更小的子问题，相比较于「分治思想」而言，它 没有「合并」的过程。

递归与分治

由于递归和分治大多数都是"孟不离焦，焦不离孟"的状态，所以很多初学者一开始都无法区分递归和分治，但是其实稍微多做点题两者的区别就很清晰了。

  • 递归是一种编程技巧，一种解决问题的思维方式。
  • 分治算法很大程度上是基于递归的，解决更具体问题的算法思想。
